## 养龙虾，比养娃成本还高
模型 coding 能力的提升速度，比大多数人预期的快得多。

模型能力的跃迁，带来了应用层的结构性变化。OpenRouter 每周处理 Token 量到了万亿级，消耗榜前几位——OpenClaw、Kilo Code、Claude Code——清一色是 Agent 应用 [^3]。

行业押注也在跟进。Anthropic 发布 Claude Cowork（2026 年 1 月），定位通用知识工作 Agent。OpenAI 用 GDPVal 基准量化 Agent 价值——GPT-5.2 在 44 个职业里，约 70.9% 的任务上表现优于或等同于有 14 年以上经验的人类专家 [^5]。用经济学指标衡量 Agent，这是方向上的变化。

然后撞上了"最后一公里"。

傅盛写过"AI 龙虾养成日记"，记录用 OpenClaw 搭 Agent 群的经历：Agent 确实能 7×24 小时跑，但前期要花大量精力调skill、调工具 [^6]。保姆式调教成本大，还有**这个过程不可复制**，你为某个电商场景磨出来的，换个类目就不好用了。每个业务场景，从零开始，成本线性增长。

"**Agent 训练场**"，是一种工程方案：给定一个业务场景和可量化的验收标准，通过自动化试错循环把通用 Agent 适配到这个场景，最后以可读、可改的 Skill 固化下来。

## 电商商家新品 Agent
"定时帮我发一条小红书" 也不简单 —— 输入选题想法，需要 Agent：搜索抓取内容，生成图片和文案，调整成你的品牌风格，最后用无头浏览器完成发布操作。这个过程也是需要多步和推理的，**但它是 "开放领域"、"无对抗"，随着 skill 市场积累，需要自己从头调试是会越来越少**。

但"电商商家新品 Agent"不一样。它是一个**符合企业要求、对抗循环**的运营过程，不是一次线性执行就结束的：

```plain
                    ┌─────────────────────────────┐
                    │                             │
                    ▼                             │
         竞品监控 & 市场感知                       │
         （类目动态、定价、销量趋势）               │
                    │                             │
                    ▼                             │
         选品 & 策略制定                           │
         （结合企业库存、利润目标、历史爆品数据）    │
                    │                             │
                    ▼                             │
         内容生成                                  │
         （商品描述 + 营销主图，匹配品牌调性）       │
                    │                             │
                    ▼                             │
         上架发布                                  │
         （商家后台操作、价格设置、分类归属）        │
                    │                             │
                    ▼                             │
         实时销售监控  ─── 不达标 ──→ 策略调整 & 寻源优化
         （曝光/转化/破零情况）        （调价、1688比价、换素材）
                    │                             │
                    └─────────── 达标 ────────────┘
                                  │
                              下一个新品
```

这个循环的关键不是"多了几个步骤"，而是每一轮的决策依赖**企业内部数据**——你的利润线在哪里，哪些品类是战略重点，哪些供应商有优先合作协议。这些东西，通用 Agent 完全不知道。

验收标准只有两个数字：每月自动发布新品 ≥ 50 个，7 天内"破零"率 > 60%。不是"Agent 走了多少步"，是人真正关心的结果。

## 企业 Agent 训练场
在搭训练场之前，有一个更基本的问题：为什么不能直接用通用 Agent，而一定要搭自己的训练场？

**企业 Agent 的核心价值，来自它掌握的企业内部知识和真实反馈信号**。这两样东西，通用 Agent 天然没有。

如果你只是给通用 Agent 一个"帮我发布新品" 命令，它确实能干活。但它优化的目标，是"完成任务"，不是"符合你的运营逻辑"。

**企业 Agent 训练场是必须的，因为：**

1. **企业内部要求不知道**。通用 Agent 是不可能知道，企业内一堆运营规则，让 Agent 把这些知识内化到行为模式里。
2. **奖励信号必须来自真实业务反馈**。"7日破零率 > 60%"这个标准，只有接入你自己的销售系统才能量化。用历史真实数据回放训练，Agent 才能学到真能力。
3. **业务要求是动态变化的**。竞品格局变了，平台规则变了，你的品类重心变了 —— 训练场可以持续接入新数据重新迭代。

这就是训练场存在的根本理由：**是为了训练出真正懂你生意的 Agent**。

## 训练场的设想
> 还是一些非常初级的想法，有团队未来做这个方向的、联系我
>

### 第一层：容器化练兵场地
Agent 在训练过程中不可预测——可能访问错误的文件，调用错误的 API，甚至尝试修改自己的运行环境。跑在宿主机上，副作用互相污染，失败也没法复现。

NanoClaw 的安全模型直接说明了正确做法：

> "When you're building with AI agents, they should be treated as untrusted and potentially malicious... Security has to be enforced outside the agentic surface, not depend on the agent behaving correctly."
>

每次任务调用起一个新容器，跑完销毁。Agent 作为非特权用户运行，只能访问显式挂载进来的目录。容器边界由 OS 来保证，不依赖应用层的 allowlist 或确认提示。

但光有隔离还不够，还需要"够真实"。训练"电商商家新品 Agent"时，沙箱需要包含：电商后台 API 接口、竞品数据源、比价寻源工具、图像生成能力、销售数据反馈管道，以及——最关键的——**企业内部数据**（历史订单、利润结构、供应商优先级）。

### 第二层：结果导向的奖励信号，Agent RL
这里有个常见陷阱：用过程指标打分，而不是结果指标。

+ ❌ **过程指标**：Agent 成功调用了竞品监控 API
+ ✅ **结果指标**：新品在 7 天内破零了，且成本低于人工推广

过程指标容易训练出"走通流程但业务价值为零"的 Agent。结果指标直接挂 KPI，这才是 Google Cloud Agent 评估方法论的核心原则：

> "Generic goals like 'be helpful' must translate to concrete statements—for example, a booking agent's success is 'correctly booking multi-leg flights meeting all user constraints.'"
>

反馈闭环跑四步：**Detect**（自动评估验收标准）→ **Diagnose**（回放轨迹）→ **Decide**（用累积评分做晋级门控）→ **Deploy**（带可观测性推送，继续收真实反馈）。

要特别注意"静默失败"（silent failures）——Agent 走了完全错误的路径，但产出了表面合理的结果。只有结果导向的评估才能识别这类问题 [^9]。

### 第三层：训练成果固化成 Skills
这是整个架构里最后一层。

NanoClaw 的做法是这样的：

> "New functionality comes through skills: instructions with a full working reference implementation that a coding agent merges into your codebase... Every installation ends up as a few thousand lines of code tailored to the owner's exact requirements."
>

工作流：

```plain
创建 Skill → 安装/应用 skill → 生成真实代码 → 确定性运行 → 可继续魔改
```

每个环节的含义：

+ **创建 Skill**：在 `.claude/skills/` 下写"代码改造说明"（`SKILL.md` + 参考模板）。把训练中摸索出的最佳实践，写成结构化的代码变更指令。
+ **安装/应用**：`skills-engine` 对仓库源码打补丁——加文件、改路由、加依赖。不是改配置，是改代码。
+ **生成真实代码**：Skill 执行完，所有变更都在仓库里。运行时是普通 Node 项目 + 容器 Agent SDK，行为由代码决定，没有动态解释配置的步骤。
+ **可继续魔改**：Skill 只是起点。后续可以像任何项目一样修改、重构，也可以叠加更多 Skill。

## 全流程串起来
三层拼在一起，"电商商家新品 Agent"的诞生流程：

```plain
步骤一：业务同学定义
  功能：竞品监控 → 选品策略 → 内容生成 → 上架发布 → 实时监控 → 循环优化
  验收：月发布 ≥ 50个，7日破零率 > 60%

步骤二：训练场自动循环（含企业内部数据）
  环境：容器化沙箱 + 电商后台API + 竞品数据源 + 比价寻源工具
        + 图像生成 + 销售反馈 + 企业利润/库存/供应商数据
  循环：通用Agent执行 → 记录轨迹 → 评估验收标准
        → 标注失败 → 更新策略 → 再执行

步骤三：Skills 输出（代码，不是Prompt）
  add-product-monitor       竞品监控 & 选品
  add-listing-publisher     内容生成 & 上架发布
  add-sourcing-optimizer    比价寻源 & 供应链优化
  add-zero-break-tracker    实时监控 & 循环反馈

步骤四：部署与迭代
  安装Skills → 确定性代码 → 部署
  根据真实业务反馈持续迭代
```

## 几个真实问题
**训练环境不够真实，Skill 在生产里会失效吗？**

会，这是真实挑战，**符合企业要求、真实对抗训练** 才是有用的

**每个场景都搭一套训练场，总成本不是更高？**

训练场的基础设施（容器化沙箱、评估框架、Skills 打包机制）是复用的，变的只是验收标准和工具集。

**Skill 生成的代码质量怎么保证？**
